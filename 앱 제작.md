# 강동어울림복지관 "이용상담문의" 게시판 실시간 모니터링 & 구글챗 알림 앱

> 이 문서는 AI 코딩 에이전트(Claude Code 등)가 이 문서만 읽고도 완전히 동작하는 앱을 처음부터 끝까지 구현할 수 있도록 작성된 기획/설계 명세서입니다. 모호한 부분 없이 아키텍처, 데이터 구조, 로직, UI 스타일까지 명시합니다.

## 1. 왜 이 아키텍처인가 (중요 — 반드시 이 구조를 따를 것)

GitHub Pages는 **정적 파일만 호스팅**하며 서버 코드나 크론(cron) 작업을 실행할 수 없습니다. 하지만 이 앱은 다음이 반드시 필요합니다:

- 주기적으로(예: 3~5분마다) 게시판 HTML을 가져와 새 글을 감지하는 **서버 측 폴링**
- 웹훅 URL, 비밀번호 등 **민감정보의 서버 측 저장**
- 대한민국 시간(KST) 기준의 **요일/시간 조건부 발송 스케줄링**

따라서 아래와 같이 역할을 분리합니다.

```
┌─────────────────────────┐        HTTPS (fetch)        ┌──────────────────────────────┐
│   GitHub Pages (정적)    │ ───────────────────────────▶ │  Google Apps Script Web App   │
│  - 로그인 화면            │ ◀─────────────────────────── │  (doGet / doPost API 역할)     │
│  - 관리자 대시보드         │        JSON 응답             │  - 게시판 크롤링 (시간기반 트리거) │
│  - index.html / css / js │                              │  - 발송 로직 / 큐 관리          │
└─────────────────────────┘                              │  - 구글챗 웹훅 발송              │
                                                            │  - 데이터 저장(Google Sheet)   │
                                                            └──────────────────────────────┘
```

- **프론트엔드(GitHub Pages)**: 로그인 화면, 대시보드 UI만 담당. 순수 정적 HTML/CSS/JS(React 불필요, Vanilla JS 권장 — 빌드 과정 없이 GitHub Pages에 바로 배포 가능해야 함).
- **백엔드(Google Apps Script)**: 실제 크롤링, 스케줄링, 발송, 데이터 저장을 전담. `doGet(e)` / `doPost(e)`로 REST API처럼 동작시키고, Web App으로 배포(`실행: 나`, `액세스: 모든 사용자`)하여 GitHub Pages에서 `fetch()`로 호출.
- **데이터 저장소**: 별도 DB 없이 **Google Sheets 1개**를 DB처럼 사용(아래 4장 스키마 참조). 민감정보(웹훅 URL, 비밀번호 해시)는 `PropertiesService.getScriptProperties()`에 저장하는 것을 권장(시트보다 접근이 제한적).
- **스케줄러**: Apps Script의 **시간 기반 트리거(Time-driven Trigger)**를 5분 간격으로 설정하여 크롤링 + 발송 로직을 주기 실행.

> AI 구현 시 주의: React/Vue 등 빌드 도구를 쓰면 GitHub Pages(root 배포) 설정이 복잡해지므로, 특별한 요청이 없는 한 **순수 HTML/CSS/JS(Vanilla)**로 구현할 것.

## 2. 핵심 기능 상세

### 2.1 게시판 실시간 모니터링

**동작 방식**
1. Apps Script 시간 기반 트리거가 **5분마다** `checkNewPosts()` 함수를 실행.
2. `UrlFetchApp.fetch(모니터링URL)`로 게시판 목록 페이지 HTML을 가져옴.
3. HTML을 파싱하여 게시글 목록(제목, 작성자, 작성일시, 게시글 고유 번호 또는 URL)을 추출.
   - 파싱은 정규식 또는 `XmlService`를 활용. 게시판 마크업 구조는 사이트마다 다르므로, **대시보드에서 관리자가 CSS 선택자 또는 목록 행 패턴을 지정할 수 있는 "파싱 규칙" 입력란**을 두거나, 최초 구현 시 강동어울림복지관 게시판의 실제 HTML 구조를 확인하여 하드코딩. (구현 착수 전 AI는 실제 URL을 가져와 HTML 구조를 먼저 분석해야 함)
4. 추출한 게시글의 **고유 식별자(게시글 번호 or 상세페이지 URL)**를 Google Sheet의 `PostLog` 시트에 이미 기록된 목록과 비교.
5. 새 게시글이면 `PostLog`에 기록하고, 발송 로직(2.2)으로 전달.

**신규 글 판별 기준**: 게시글 고유 번호(URL 파라미터의 `no`, `idx` 등) 기준. 제목 텍스트만으로 판별하지 말 것(중복/수정 오탐 방지).

### 2.2 시간대별 발송 로직 (핵심 비즈니스 로직)

**규칙 요약**
- 발송 가능 시간대: **평일(월~금) 09:00 ~ 18:00 (KST, `Asia/Seoul`)**
- 이 시간대에 감지된 새 글 → **즉시 구글챗 발송**
- 이 시간대 밖(평일 18:00 이후 09:00 이전, 토요일, 일요일)에 감지된 새 글 → **큐(Queue)에 저장**, 발송하지 않음
- 큐에 쌓인 글은 **다음 발송 가능 시각의 09:00 정각**에 일괄 발송

**일반화된 로직 (예시 1, 2를 포함하는 일반 규칙)**

```
IF 오늘이 월~금 AND 현재시각이 09:00~18:00 이내:
    즉시 발송
ELSE:
    큐에 저장 (발송 보류)
    "다음 09:00" 트리거 시점에 큐를 비우며 일괄 발송

"다음 09:00"의 정의:
    - 지금이 평일 18:00 이후, 다음날이 평일이면 → 다음날 09:00
    - 지금이 금요일 18:00 이후 / 토요일 / 일요일 09:00 이전 → 다음 월요일 09:00
    - (공휴일 처리는 1차 버전 범위 밖. 필요 시 관리자가 대시보드에서 "임시 휴무일" 등록하는 기능으로 확장 가능하도록 설계는 열어둘 것)
```

**구현 방법 (Apps Script)**
- 시간 기반 트리거는 5분마다 `checkNewPosts()`와 `flushQueueIfDue()` 두 함수를 함께 실행.
- `flushQueueIfDue()`는 다음을 체크:
  ```
  now = 현재시각을 Asia/Seoul 기준으로 변환
  IF now.요일 IN [월,화,수,목,금] AND now.시 == 9 AND now.분 < 5 (트리거 주기 내 1회만 실행되도록):
      큐에 있는 모든 미발송 게시글을 하나의 메시지(또는 순차 메시지)로 구글챗 발송
      큐 비우기 / 발송 상태 업데이트
  ```
- 09:00 트리거 시점에 이미 5분 폴링 주기로 "지금 막 들어온 글"과 "큐에 있던 글"이 섞이지 않도록, flush 로직에서는 **큐 테이블에 있는 기존 항목만** 처리하고 신규 크롤링 결과는 별도로 이 판단 로직(2.2 규칙)을 다시 통과시킬 것.
- 09:00~18:00 사이라도 "지금 막 크롤링에서 발견됐지만 사실은 18:00 이후 작성된 글"이 있을 수 있음(폴링 주기 지연). 이 경우 **게시글의 실제 작성시각**(게시판에 표시된 작성일시)을 기준으로 판단하는 것이 원칙. 단, 게시판에 "시:분"까지 표시되지 않는 경우 감지 시각(폴링 시각)을 대체 기준으로 사용.

### 2.3 구글챗(Google Chat) 발송

- 대시보드에서 등록한 **Webhook URL**로 `UrlFetchApp.fetch(webhookUrl, {method:'post', contentType:'application/json', payload: JSON.stringify(message)})` 호출.
- 메시지 포맷 예시(Google Chat Card 또는 단순 text):
  ```json
  {
    "text": "📩 새로운 이용상담문의 게시글이 등록되었습니다.\n\n제목: {title}\n작성자: {author}\n작성일시: {postedAt}\n링크: {url}"
  }
  ```
- 큐 일괄 발송 시에는 게시글 개수만큼 반복 발송하거나, 하나의 메시지에 리스트로 묶어서 발송(권장: 메시지 여러 개보다 **하나의 카드에 목록으로 묶어서 발송** — 예: "월요일 오전 9시 알림 — 주말 사이 접수된 상담글 3건" 후 목록 나열).
- 발송 실패 시(웹훅 URL 오류, 네트워크 오류) 재시도 로직: 최대 3회, 각 시도 사이 지수 백오프(예: 1분, 5분, 15분 뒤 재시도) 또는 다음 폴링 주기에 재시도. 실패 로그는 시트에 기록.

## 3. 관리자 인증 & 대시보드

### 3.1 로그인
- 초기 계정: **아이디 `admin` / 비밀번호 `1234`**
- 최초 배포 시 Apps Script `PropertiesService`에 `ADMIN_ID = "admin"`, `ADMIN_PW_HASH = SHA256("1234")` 형태로 저장(평문 저장 금지, 해시로 저장).
- 로그인 API: `POST /login` (Apps Script `doPost`에서 `action=login` 파라미터로 라우팅) → 아이디/비번 검증 → 성공 시 세션 토큰(랜덤 문자열, UUID) 발급, `PropertiesService` 또는 시트에 토큰과 만료시각(예: 발급 후 12시간) 저장.
- 프론트엔드는 발급받은 토큰을 `localStorage`(GitHub Pages는 서버가 없으므로 세션 관리는 클라이언트 localStorage + 매 요청마다 토큰 검증하는 방식)에 저장하고, 이후 대시보드 API 호출 시 토큰을 함께 전송.
- 토큰 만료/미존재 시 로그인 화면으로 리다이렉트.

### 3.2 비밀번호 변경 기능
- 대시보드 내 "비밀번호 변경" 폼: 현재 비밀번호, 새 비밀번호, 새 비밀번호 확인 3개 입력란.
- API: `POST /changePassword` → 현재 비밀번호 해시 일치 확인 → 일치 시 `ADMIN_PW_HASH`를 새 비밀번호 해시로 갱신.
- 프론트엔드 유효성 검사: 새 비밀번호 최소 4자리 이상, 새 비밀번호와 확인값 일치 여부.

### 3.3 대시보드 기능

**(1) 모니터링 URL 관리**
- 복지관 홈페이지 "이용상담문의" 게시판 목록 URL을 등록/수정/삭제.
- 여러 개 등록 가능하도록 설계(향후 다른 게시판 확장 대비), 단 1차 버전은 최소 1개 필수 등록.
- 입력값: `label`(표시용 이름), `url`, `enabled`(모니터링 on/off 토글).
- CRUD API: `GET /urls`, `POST /urls`, `PUT /urls/:id`, `DELETE /urls/:id`

**(2) 구글챗 웹훅 URL 관리**
- 구글챗 Space의 Incoming Webhook URL을 등록/수정/삭제.
- CRUD API: `GET /webhooks`, `POST /webhooks`, `PUT /webhooks/:id`, `DELETE /webhooks/:id`
- 웹훅 URL은 화면에 전체 노출하지 않고 마스킹 처리(예: `https://chat.googleapis.com/v1/spaces/AAA...***`) 후 "보기"/"수정" 클릭 시에만 전체 표시.

**(3) 웹훅 연동 테스트 기능**
- "테스트 발송" 버튼 클릭 시 등록된 웹훅으로 샘플 메시지(`"✅ 테스트 메시지입니다. 연동이 정상적으로 완료되었습니다."`) 즉시 발송.
- API: `POST /webhooks/:id/test`
- 성공/실패 결과를 대시보드에 토스트 알림으로 표시.

**(4) 모니터링 현황 (권장 추가 기능 — 없으면 관리자가 정상 동작 여부를 알 수 없으므로 포함 권장)**
- 최근 감지된 게시글 목록(발송완료/대기중 상태 표시)
- 마지막 폴링 시각, 다음 발송 예정 시각(큐 대기 중일 경우) 표시
- API: `GET /status`, `GET /logs`

## 5. API 명세 (Google Apps Script Web App 기준)

Apps Script는 하나의 URL(`/exec`)만 가지므로, 쿼리 파라미터 `action`으로 라우팅합니다.

| 목적 | Method | action 값 | 필요 파라미터 | 인증 |
|---|---|---|---|---|
| 로그인 | POST | login | id, pw | X |
| 비밀번호 변경 | POST | changePassword | token, currentPw, newPw | O |
| 모니터링 URL 목록 조회 | GET | listUrls | token | O |
| 모니터링 URL 등록 | POST | addUrl | token, label, url | O |
| 모니터링 URL 수정 | POST | updateUrl | token, id, label, url, enabled | O |
| 모니터링 URL 삭제 | POST | deleteUrl | token, id | O |
| 웹훅 목록 조회 | GET | listWebhooks | token | O |
| 웹훅 등록 | POST | addWebhook | token, label, webhookUrl | O |
| 웹훅 수정 | POST | updateWebhook | token, id, label, webhookUrl | O |
| 웹훅 삭제 | POST | deleteWebhook | token, id | O |
| 웹훅 테스트 발송 | POST | testWebhook | token, id | O |
| 상태 조회(대시보드용) | GET | status | token | O |
| 최근 로그 조회 | GET | logs | token | O |

- 인증(O) 항목은 서버에서 `token`이 `Settings`에 저장된 값과 일치하고 만료되지 않았는지 검증 후 처리.
- 모든 응답은 JSON: `{ "success": true/false, "data": ..., "message": "..." }`
- CORS: Apps Script Web App은 기본적으로 CORS 이슈가 있을 수 있으므로, `doGet`/`doPost`에서 `ContentService`로 JSON 응답 시 `setMimeType(ContentService.MimeType.JSON)` 사용하고, 프론트엔드 `fetch`는 GitHub Pages 도메인에서 호출됨을 감안하여 Apps Script 배포 시 "액세스 권한: 모든 사용자"로 설정.

**크롤링/발송 관련 함수 (트리거 전용, 외부 API 아님)**
- `checkNewPosts()` — 5분 주기 트리거
- `flushQueueIfDue()` — 5분 주기 트리거 (checkNewPosts와 같은 트리거에서 순차 호출해도 무방)
- `sendToGoogleChat(message, webhookUrl)` — 내부 함수

### 6.3 폰트: Paperlogy
- Paperlogy는 웹폰트 CDN(예: jsdelivr 기반 webfontworld 배포)으로 제공되는 한글 폰트입니다. AI 구현 시 최신 CDN 링크를 직접 검색/확인하여 `@font-face` 또는 `<link>`로 적용할 것(문서 작성 시점의 링크가 변경/만료될 수 있으므로 구현 단계에서 재확인 필요).
- 참고 적용 예시:
  ```css
  @font-face {
    font-family: 'Paperlogy';
    src: url('https://cdn.jsdelivr.net/gh/webfontworld/paperlogy/Paperlogy-Regular.woff2') format('woff2');
    font-weight: 400;
    font-display: swap;
  }
  @font-face {
    font-family: 'Paperlogy';
    src: url('https://cdn.jsdelivr.net/gh/webfontworld/paperlogy/Paperlogy-Bold.woff2') format('woff2');
    font-weight: 700;
    font-display: swap;
  }
  body { font-family: 'Paperlogy', -apple-system, sans-serif; }
  ```
- CDN 접근이 안 될 경우를 대비해 시스템 폰트(`-apple-system, "Apple SD Gothic Neo", "Malgun Gothic", sans-serif`)를 fallback으로 반드시 지정.

## 7. GitHub 배포 안내

1. 리포지토리 루트에 `index.html`(로그인), `dashboard.html`(대시보드), `/assets/css`, `/assets/js` 구조로 정적 파일 배치.
2. GitHub 저장소 Settings → Pages → **Source: Deploy from a branch → Branch: `main` / Folder: `/(root)`** 로 설정.
3. Apps Script Web App 배포 URL(`https://script.google.com/macros/s/xxxxx/exec`)을 프론트엔드 JS 상단에 상수로 지정(`const API_BASE = "..."`).
4. 배포 후 GitHub Pages URL로 접속하여 로그인 → 대시보드 → 모니터링 URL/웹훅 등록 → 테스트 발송까지 확인.

### 권장 폴더 구조
```
/
├── index.html          (로그인)
├── dashboard.html       (관리자 대시보드)
├── assets/
│   ├── css/
│   │   └── style.css    (글래스모피즘 공통 스타일)
│   └── js/
│       ├── api.js       (API_BASE 상수 + fetch 래퍼)
│       ├── auth.js       (로그인/토큰 관리)
│       └── dashboard.js  (CRUD, 테스트발송 로직)
└── 앱 제작.md            (본 명세서)
```

## 9. 구현 우선순위 (AI가 순서대로 작업할 것)

1. Google Apps Script 프로젝트 생성 + Google Sheet(DB) 생성 및 4장 스키마대로 시트 구성

6. `sendToGoogleChat()` 구현 + 테스트 발송 API 연결
7. 프론트엔드 로그인 페이지(`index.html`) 구현
8. 프론트엔드 대시보드(`dashboard.html`) 구현 — URL관리/웹훅관리/테스트발송/비밀번호변경/로그조회
9. 글래스모피즘 스타일 + Paperlogy 폰트 적용 (6장 가이드 기준)
10. GitHub Pages 배포 설정 및 End-to-End 테스트 (평일 09~18시 즉시발송 케이스, 평일 18시 이후 큐잉→익일 09시 발송 케이스, 주말 큐잉→월요일 09시 발송 케이스 3가지 시나리오 반드시 검증)

## 10. 보안 체크리스트

- [ ] 비밀번호는 평문 저장 금지, SHA-256 이상 해시로 저장
- [ ] 구글챗 웹훅 URL은 프론트엔드 소스코드에 하드코딩하지 말고 반드시 백엔드(Apps Script Properties/Sheet)에만 저장

- [ ] Apps Script Web App 배포 시 실행 계정은 "나"로, 액세스는 "모든 사용자"로 하되, 모든 쓰기성 action은 token 검증 필수
